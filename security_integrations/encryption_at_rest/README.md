YugabyteDB Anywhere (YBA) needs a Key Management Service (KMS) for Encryption at Rest (EaR) because of how its **envelope encryption** key hierarchy works:

1. **Data Encryption Keys (DEK)** — Symmetric keys that directly encrypt the data files. Each file flushed from memory gets a unique DEK, generated by YugabyteDB itself.
2. **Universe Key** — A symmetric key that encrypts and decrypts all the DEKs in a universe. Generated by YBA.
3. **Master Key** — The top-level key that encrypts the universe keys. This is a **Customer Managed Key (CMK)** stored and managed in a KMS.

The KMS is required specifically to **store and manage the master key**. YBA never stores the master key itself — it only stores the KMS configuration (i.e., a reference to where the master key lives). When data needs to be encrypted or decrypted, YBA uses the KMS to access the master key, which in turn unlocks the universe keys, which in turn unlock the DEKs. [[Encryption at rest](https://docs.yugabyte.com/stable/yugabyte-platform/security/enable-encryption-at-rest/)]

This design means:
- The most sensitive key (the master key) is never directly held by YBA.
- Access to all encrypted data ultimately depends on access to the KMS.
- If the KMS is revoked or the master key is deleted, the data remains protected even if someone gains access to the database files.

YBA supports CMKs in **AWS KMS, GCP KMS, Azure Key Vault, and HashiCorp Vault**.

## Install and Configure Hashicorp Vault

Before we create a KMS confdiguration in YBA for HashiCorp Vault, we need to install and configure Vault.

### Install Vault

Follow the [official instructions](https://developer.hashicorp.com/vault/install#linux) to install Vault on a RHEL VM.

### Configure Vault 

#### TLS configuration

Check the default configuration at `/etc/vault.d/vault.hcl`. It should look like the following:

```
storage "file" {
  path = "/opt/vault/data"
}

# HTTPS listener
listener "tcp" {
  address       = "0.0.0.0:8200"
  tls_cert_file = "/opt/vault/tls/tls.crt"
  tls_key_file  = "/opt/vault/tls/tls.key"
}
```

#### Generate Certs

Create a file named `vault-cert.cnf` in your working directory and update the contents with the following:

```ini
[req]
distinguished_name = req_distinguished_name
req_extensions     = v3_req
prompt             = no

[req_distinguished_name]
# The Common Name (CN) isn't strictly used for validation anymore, but is good for labeling.
CN = vault.local

[v3_req]
subjectAltName = @alt_names

[alt_names]
IP.1  = 127.0.0.1
IP.2  = 10.0.0.50      # <-- UPDATE THIS to your Vault Server IP
DNS.1 = vault.local    # <-- UPDATE THIS to your Vault DNS name (if you have one)
DNS.2 = localhost

```

Ensure you have set the correct IP address and/or DNS for the server running Vault.

Then, run the [vault_certificate_generator.bash script](./vault_certificate_generator.bash) to generate the Root CA and key, along with the server cert and key. Alternatively, you may run each step of the script directly in bash.

```bash
# Ensure the script is executable
./vault_certificate_generator.bash
```

Move `tls.crt` and `tls.key` to `/opt/vault/tls/` on your Vault server, replacing the existing certificates there. Ensure the 'vault' user has read permissions for these files. 


Finally, restart Vault so that it uses the new certificates.
```
sudo systemctl restart vault
sudo systemctl status vault
```


#### Initialize & Unseal

To use Vault CLI, set the following:
`export VAULT_CACERT="/path/to/your/root-ca.pem"`

For ease of unsealing the Vault for the POC environment, we can initialize Vault to only require one unseal key instead of the default three.

```bash
# Initialize and output the keys in JSON format
vault operator init -key-shares=1 -key-threshold=1 -format=json > vault-init.json
```

Optional: Create a script to unseal
```
#!/bin/bash
# Extract the single unseal key using jq and unseal Vault
export VAULT_ADDR="http://127.0.0.1:8200"
UNSEAL_KEY=$(jq -r ".unseal_keys_b64[0]" vault-init.json)

vault operator unseal $UNSEAL_KEY
echo "Vault unsealed successfully!"
```

Unseal Vault:
```
vault status

vault operator unseal
```

#### Enable secrets engine

Enable the secret engine by executing the following command:
```
vault secrets enable transit
```

#### Create Vault Policy

```bash
# Create the policy file
cat << 'EOF' > yb-ear-policy.hcl
path "transit/*" {
  capabilities = ["create", "read", "update", "delete", "list"]
}

path "auth/token/lookup-self" {
        capabilities = ["read"]
}

path "sys/capabilities-self" {
        capabilities = ["read", "update"]
}

path "auth/token/renew-self" {
        capabilities = ["update"]
}

path "sys/*" {
        capabilities = ["read"]
}
EOF

# Write policy
vault policy write yb-ear-policy yb-ear-policy.hcl

# Verify it shows up in the list
vault policy list
```

#### Create a Token

```
vault token create -no-default-policy -orphan -period="768h" -policy="yb-ear-policy"
```
## Create a KMS configuration in YBA

Before creating the KMS configuration in YBA, it is essential to ensure YBA trusts Vault.

Add the root CA certificate into YBA's trust store by following [these steps](https://docs.yugabyte.com/stable/yugabyte-platform/security/enable-encryption-in-transit/trust-store/#add-certificates-to-your-trust-store).

Follow the [instructions to create a KMS configuration](https://docs.yugabyte.com/stable/yugabyte-platform/security/create-kms-config/hashicorp-kms/#create-a-kms-configuration). Use _Token_ as the _Authentication Type_, providing the details obtained in previous steps.


## Enable encryption at rest

Enable encryption at rest on an existing universe by following [these instructions](https://docs.yugabyte.com/stable/yugabyte-platform/security/enable-encryption-at-rest/#enable-encryption-at-rest-on-an-existing-universe).
